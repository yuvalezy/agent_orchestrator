# M1.7 — Service-desk ticket ingestion (blueprint)

**Milestone:** M1.7 (change 01 §3.6, design D6). **Repo:** `/mnt/dev/tools/agent_orchestrator`.
**Depends on:** M0 (change-00 `updatedAfter` ticket filter + thread-touch) — **already live in dev** (portal-business runs `feat/change-00-portal-sync-webhooks`; module migrations auto-apply on boot). Reuses the M1.5b money-loop **unchanged**.

## 1. Scope — what ships

A **poll-only** service-desk channel: the orchestrator polls the portal's ticket list on an `updatedAfter` cursor, turns **new tickets** and **new public customer replies** into `agent_inbox` rows (`thread_key = ticket id`), and lets the existing money-loop create a task per ticket / comment per reply. New ticket → task; customer reply → comment; all via the **same** dedup path (source triple `service_desk` + ticket id).

**In scope (read/ingestion only):**
- `EzyPortalGateway` gains `implements TicketingPort` — `listChangedTickets` (paginated drain) + `getThread` (public entries).
- `ServiceDeskAdapter implements ChannelAdapter` — maps ticket + new public thread-entries → `InboundMessage`, with skip rules.
- A **generic** reconcile worker (extracted from the M1.6 email worker — DRY) drives both email and service-desk.
- Registry wires the seeded `ezy_service_desk` instance; `main.ts` starts one SD poller.
- Core contact-resolution gains a `service_desk` branch (resolve customer by the **backfilled BP ref** primary, then the existing email contact lookup + domain-propose fallback) — the **only** core touch.

**Explicitly OUT of scope (deferred):**
- **Outbound** (`postReply`, `setStatus`) — needs `service-desk.manage`; implemented on the gateway but **not wired** (M1.8 / Phase-4). Read path needs only `service-desk.view`.
- **Webhook push** (`service-desk.ticket.*` events) — the poll path is the baseline; push is a later upgrade (D6).
- Ticket resolution → task-status sync (Phase 4).

## 2. Ground truth (from portal recon — cite before building)

Go routes are under `/api`; through nginx the orchestrator calls `/api/service-desk/*` (CLAUDE.md rewrite rule). All ticket reads require **`service-desk.view` @ Read** on the tenant API key (a no-group key = full access; we use the scoped group + add this permission).

| Need | Endpoint (via nginx) | Shape / notes |
|---|---|---|
| Changed tickets | `GET /api/service-desk/tickets?updatedAfter=<RFC3339>&sortBy=updatedAt&sortDescending=false&pageNumber=&pageSize=100` | Envelope `{data, totalCount, pageNumber, pageSize}`. `updatedAfter` is **INCLUSIVE `>=`**. Tiebreak is always `ticket_number DESC`. `pageSize` hard-capped at 100. No `status` filter passed → all statuses returned (incl. `resolved`/`closed`). Tenant is implicit from the key. |
| One ticket | `GET /api/service-desk/tickets/:id` | Raw `Ticket`. `:id` = the **UUID** from the list (also accepts ticket number). |
| Thread (replies) | `GET /api/service-desk/tickets/:id/thread?visibility=public` | Raw **array** of `TicketThreadEntry`, `created_at ASC`, **no time filter** → diff client-side by `entry.id`. `?visibility=public` excludes internal notes. |

**`Ticket`:** `id`(uuid), `ticketNumber`(`SD-00042`), `subject`(required), `description`(nullable — **the initial body**), `status`(`open|pending|resolved|closed`; terminal = `resolved`+`closed`), `priority`(`low|medium|high|urgent`), `source`, `requesterType`(`bp|account|manual`), `requesterBPID`(nullable uuid — **BP link; trust only when `requesterType=='bp'`**), `requesterContactID`, `requesterName`, `requesterEmail`(nullable), `accountId`, `createdAt`, `updatedAt`, `tags`.

**`TicketThreadEntry`:** `id`(uuid), `ticketId`, `entryType`(`reply|internal_note|system`), `visibility`(`public|internal`), `body`, `authorName`, `authorIsExternal`(**true = customer, false = staff/agent**), `createdAt`.

**Confirmed absent:** no time-filtered thread endpoint; no per-tenant thread listing; no delete tombstone (soft-delete = silent disappearance); no `tenantId` query param.

## 3. Key decisions (DA: challenge these)

**D-A — Customer resolution: BP-ref primary (now backfilled), email fallback (the one core touch).** ⚠ **Re-verified after a portal change (2026-07-06).** Yuval modified service-desk ticket creation to **backfill `requesterBPID` + `requesterContactID` from `requesterUserID`** for external-account tickets. Verified live: `SD-00001` **and** `SD-00002` are `requesterType='account'` but now carry **`requesterBPID = 5cc23a0f…`, which exactly equals the Test Customer's `agent_customers.bp_ref`**. So the BP link is now an authoritative, exact-match resolver key (no fuzzy email/domain needed). This **supersedes the earlier email-only correction** (which was made when `requesterBPID` was NULL); email stays as the fallback for any ticket the backfill doesn't cover (e.g. `manual` tickets, or an account with no linked BP).

**Every emitted `InboundMessage` — the initial ticket message AND every reply entry — sets `sender.address = (ticket.requesterBPID ?? ticket.requesterEmail ?? '').toLowerCase()`** (B1). This is critical: `TicketThreadEntry` carries **no author identity we can resolve**, and `resolveContact` keys solely on `{channelType, address}`; a reply emitted with an empty/author-derived address would resolve `unknown` → skip → **gate step 2 (reply→comment) would silently fail**. The reply author *is* the ticket requester, so the whole thread shares the requester's identity (BP-ref preferred, email fallback). `displayName = requesterName`; `requesterType`/`requesterBPID`/`requesterEmail` also go in `raw` (B5).

Core change in `resolveContact`: a `service_desk` branch that tries, in order — the single `address` holds a bp-ref UUID **or** an email, and each lookup harmlessly misses the wrong type:
1. `findCustomerByBpRef(address)` → **known** (authoritative; resolves `SD-00001`/`SD-00002` by the backfilled BP link = `agent_customers.bp_ref`). **NEW** query in `ContactResolutionQueries` (`SELECT id FROM agent_customers WHERE bp_ref = $1`); returns `{customerId, contactId:''}` — only `customerId` is consumed downstream (**builder: confirm `contactId` is not load-bearing** in `triage.service.ts`/`recordTaskBridge`).
2. `findContactByAddress('email', address)` → **known** (fallback — a ticket whose `requesterBPID` is null but whose `requesterEmail` is a known email contact; e.g. the pre-backfill state).
3. email-domain **propose** — only when `address` contains `@`; an accepted contact imports as `channel_type='email'`, **not** `'service_desk'` (B4).
4. → **unknown → skip** (`bumpSkipped`/`markSkipped`). *(B5 de-scoped after verify: the skip-log enrichment existed to measure the **deferred** BP-ref gap; the backfill **closed** that gap, so it's moot. `requesterType`/`requesterBPID` still land in `agent_inbox.raw_metadata` (queryable via SQL) — not worth threading into the core claim path.)*

**B3 (still applies):** the current early-return `if (input.channelType !== 'email') return unknown` (`contact-resolution.ts:60`) sits before the domain fallback — the `service_desk` branch must be added **above** it (or the guard widened) so all four steps are reachable. Keep it minimal and **adapter-free** (D1; `lint:boundary` green). This is the robust **superset** — bp-ref (now reliable) primary, email secondary — covering backfilled account tickets AND email-only/manual tickets; **R54's deferred gap is closed.**

**D-B — Identity & threading.** One `agent_inbox` row per unit of content, all sharing the ticket as thread:
- Initial ticket message: `providerMessageId = ticket:<ticketId>`, `body = description ?? subject`, `direction=inbound`.
- Each thread entry: `providerMessageId = entry:<entryId>`, `body = entry.body`.
- **`threadKey = <ticketId>`** for every row → money-loop source triple `(agent-orchestrator, service_desk, ticketId)` groups them → **one task per ticket**, replies → comments. Dedup key in `agent_inbox` is `(channel_instance_id, channel_message_id)` → inclusive-cursor re-delivery is absorbed idempotently.

**D-C — Skip rules (avoid noise & self-loops).**
- `entryType ∈ {internal_note, system}` OR `visibility=internal` → **never ingested** (we fetch `?visibility=public`, so internal notes never arrive; belt-and-suspenders filter in the adapter).
- Public entry with `authorIsExternal=false` (staff/agent public reply, incl. our own future `postReply`) → ingest as `direction=outbound` → `ingestInbound` stores it **`skipped`** (thread context, never triaged, no send→ingest loop — mirrors email skip-self-sent).
- Public entry with `authorIsExternal=true` (customer) → `direction=inbound` → triaged.
- Initial ticket message → `direction=inbound` (a ticket is customer work). Unknown-customer tickets are gated out by D-A resolution, exactly like WA/email.

**D-D — Cursor: inclusive `>=` + drain-all + dedup-by-id.** `listChangedTickets` paginates `pageNumber` until `pageNumber*pageSize ≥ totalCount` (**drain every page before advancing** — R32 discipline). Cursor = `max(ticket.updatedAt)` across the drained set, stored in `channel_instances.sync_cursor`. Next poll passes it as `updatedAfter` → boundary rows re-delivered → deduped by `providerMessageId`. **No `cursor+1ns`** (the portal's inclusive design exists to avoid same-timestamp sibling skips; dedup-by-id is the intended pattern). The reconcile worker advances the cursor **only after every row in the tick ingests** (throw → cursor held → idempotent re-fetch).

**Accepted limitation (B7 — offset-pagination drift):** the list sorts `updated_at ASC` and the thread-touch trigger bumps `updated_at` monotonically, so if a ticket in an already-read page is bumped *mid-drain*, later rows shift toward the head and the row then sitting on a page boundary can be skipped; because the cursor advances to `max(updatedAt) > skipped.updatedAt`, the inclusive re-fetch won't recover it until that ticket next changes. This only bites when **>100 tickets change within a single drain** (essentially first-boot on a busy tenant) — there is **no livelock** (cursor is monotonic; steady-state re-fetch converges). Keyset `(updated_at, id)` pagination would be robust; **accepted at Phase-1 scale** and called out rather than hidden. Revisit if first-boot windows routinely exceed one page.

**First-run bootstrap (cursor null) — lookback window, NOT `now()`.** `updatedAfter` is required (a null cursor would list every ticket ever). On first run pass `updatedAfter = now − SERVICE_DESK_BOOTSTRAP_WINDOW_DAYS` (default **7d**), then persist `max(updatedAt)`. This mirrors the Gmail dynamic-bootstrap (R51-2) and is what lets the gate pick up the **already-created `SD-00001`** (a `now()` bootstrap would miss any ticket opened before boot). Re-scanning recent historical tickets is harmless — unresolvable ones (e.g. the `e2e@test.local` tickets) resolve `unknown` → `skipped`, never tasks. On a large tenant the window bounds first-boot volume; drain-all + pagination handle the rest.

- **B9 — empty-drain cursor:** when the first poll returns zero tickets, persist the window-start (or `now`) as the cursor, **never `null`** — otherwise every idle tick re-computes `now − 7d` and re-scans the whole sliding window forever.
- **B11 — expected first-boot activation:** any *resolvable* real-customer ticket updated within the window — **including recently-resolved ones** — will create a task on first boot (by design: no status filter). This is minor activation noise, tunable via `SERVICE_DESK_BOOTSTRAP_WINDOW_DAYS`; set it small (or 0 to start from `now`) if a clean-slate boot is wanted.

**D-E — Generic reconcile worker (DRY).** Extract the M1.6 email worker into `src/adapters/reconcile-worker.ts`: `buildReconcileWorker({instanceId, instanceName, namePrefix, fetchSince, sink, intervalMs})` where `fetchSince: (cursor) => Promise<{messages, nextCursor}>`. Re-point `buildEmailReconcileWorker` to it and use it for SD. **B6 — the extraction is proven, not asserted:** there is **no** existing `buildEmailReconcileWorker` test, so "M1.6 stays green" would be vacuous. M1.7 adds a **characterization test** for the generic worker (advance-after-all-ingest / hold-on-throw / write-only-when-`nextCursor !== cursor`); email inherits that coverage because it now routes through the same worker. *DA-resolved: the extraction is the right DRY call; the new test is what makes the refactor of already-gated code safe.*

**D-F — Gateway `TicketingPort`: read now, write later.** Implement `listChangedTickets` + `getThread` fully. Implement `postReply`/`setStatus` too (so the port is whole) but **do not wire them** — they need `service-desk.manage` and are M1.8/Phase-4. Health probe = `GET /api/service-desk/tickets?pageSize=1`.

## 6. New risks (add to RISK-REGISTER)
- **R53 — internal-note / status-churn resurfacing.** The thread-touch trigger + status edits bump `updated_at` with no new customer content → the ticket reappears in `updatedAfter`. **Mitigation:** the adapter fetches `?visibility=public` and only emits entries whose `id` is new + `authorIsExternal=true`; a resurface with no new external entry emits nothing (initial + prior entries dedup). Verify: flip a ticket's priority → no new task/comment.
- **R54 — BP-ref resolution (was deferred → now PRIMARY, gap CLOSED).** Yuval's ticket-create **backfill** populates `requesterBPID`/`requesterContactID` from `requesterUserID` for external-account tickets; verified `SD-00001`/`SD-00002` carry `requesterBPID = 5cc23a0f… = agent_customers.bp_ref`. **D-A now resolves BP-ref first** (`findCustomerByBpRef`), email second, domain-propose third. The former deferred gap (bp-typed ticket, unknown email) is closed by step 1. Residual: a `manual` ticket with neither a bp_id nor a known email → still `unknown → skip` (correct — genuinely unidentifiable). Verify: a ticket from an unknown email/no-BP → skipped, counter bumps, no task.
- **R55 — tenant-API-key 429 (localpolicy quota) + `SERVICE_DISABLED` 403.** Poller hits the shared quota / the Service-Desk app must be enabled for the tenant. **Mitigation:** http-client already retries 429 w/ Retry-After; a 403 `SERVICE_DISABLED` is a config precondition surfaced at the gate (verify Service Desk enabled on the tenant). 
- **R56 — inclusive-cursor steady re-fetch.** The newest ticket(s) re-fetch every tick until something newer bumps the cursor. **Mitigation:** idempotent upsert dedups; cost is bounded (≤ page size of same-timestamp rows). Accepted (the portal's intended pattern).
