## 1. Design decisions
### DM5a-1 · Extend the existing gateway (no new adapter)
`EzyPortalGateway` becomes `implements CustomerDirectoryPort, TaskTargetPort` (the class comment already anticipates this). Reuse the M1.2 `EzyPortalHttpClient` (`X-Api-Key`, retry on 5xx/429, **`Idempotency-Key` minted once per POST before the retry loop** — already implemented). `listWorkItemTypes` stays as-is.
### DM5a-2 · Opaque refs stay TEXT (D4)
`customerRef`/`projectRef`/`workItemTypeRef`/`task.ref` are opaque `TEXT` in/out — the gateway is the ONLY place they're known to be UUIDs. Core never parses them.
### DM5a-3 · `source*` mapping is the dedup linchpin (feeds M1.5b)
`createTask.source` → `sourceService='agent-orchestrator'`, `sourceEntityType=<channel type>`, `sourceEntityId=<thread key>`, `sourceDisplay`, `sourceUrl`. `findOpenTasks({sourceEntity:{type,id}})` → the index-backed list filter. This is exactly what M1.5b's dedup (create-vs-comment) will call — get it right here.
### DM5a-4 · setStatus via the dedicated endpoint
`POST /tasks/:id/status {status}` — NOT `PATCH /tasks/:id`. Status strings pass through opaquely (core doesn't enumerate them; portal validates). Used by change 04, built now for port-completeness + the contract test's status leg.
### DM5a-5 · No WA `/ezy-portal/*` proxy (R18)
All portal I/O goes through THIS gateway over HTTP to `:5040`; never whatsapp_manager's built-in EZY proxy, never the portal DB.
