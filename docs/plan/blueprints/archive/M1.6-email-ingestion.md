## 1. Design decisions

### DM6-1 · GmailClient (`src/adapters/email/gmail-client.ts`) implements `EmailProviderClient`
- **OAuth2:** `resolveCredential(ref)` → JSON `{client_id,client_secret,refresh_token}`; exchange for an access token (`POST https://oauth2.googleapis.com/token grant_type=refresh_token`), cache until `expires_in`-60s. All Gmail calls send `Authorization: Bearer`.
- **listChanges(cursor=historyId):**
  - **Bootstrap (cursor null / 404 expired):** `GET /gmail/v1/users/me/profile` → `historyId`; list recent inbox (`GET /messages?q=newer_than:2d in:inbox -from:me&maxResults=25`) → `messages.get` each → `ProviderEmail[]`; `nextCursor = historyId`. (Bounded — not a full backfill; that's change 03.)
  - **Incremental:** `GET /history?startHistoryId=cursor&historyTypes=messageAdded&labelId=INBOX` → dedup message ids from `messagesAdded` → `messages.get(format=full)` each → map. `nextCursor = latest historyId`. A `404` (historyId too old) → re-bootstrap.
- **messages.get → ProviderEmail:** parse headers (From/To/Cc/Subject/Message-ID/In-Reply-To/References — case-insensitive), `threadId`, `internalDate`→sentAt; **body = first `text/plain` part** (recursive MIME walk over `payload.parts`; base64url-decode; fall back to stripped `text/html` only if no plain part). Addresses parsed to bare lowercased emails.
- **getThread(threadId):** `GET /threads/{id}?format=full` → map each message → `ProviderEmail[]`.
- **send:** build an RFC-822 MIME (`To`, `Subject`, `In-Reply-To`, `References`, `Content-Type: text/plain; charset=utf-8`, body), base64url, `POST /messages/send {raw, threadId?}`. Reply headers set `In-Reply-To`/`References` from the input.
- Errors: 401 → refresh once + retry; 429/5xx → retry (reuse `adapters/shared/retry`); 4xx → surface. Never log body/token/headers.

### DM6-2 · EmailChannelAdapter (`src/adapters/email/email-channel.adapter.ts`) implements `ChannelAdapter`
Wraps ONE `EmailProviderClient` + the instance's `accountEmail`. `pull(cursor)` → `listChanges` → map `ProviderEmail`→`InboundMessage`: `instanceId`, `providerMessageId=email.id`, `threadKey=email.threadId`, `sender={address:from, displayName?}`, `recipients={to,cc}`, `direction='inbound'`, `subject`, `body=bodyText`, `replyToProviderMessageId=inReplyTo`, `raw=email`. **Skip self-sent** (`from === accountEmail` → don't yield). `send(OutboundMessage)` → `client.send` with reply headers from `threadKey`/`inReplyTo`. `health()` → a cheap `profile` call. Capabilities: threads/subjects true.

### DM6-3 · Registry + poller wiring
Channel registry (`channel-registry.ts`) gains a `gmail` provider factory → `buildEmailAdapter(instance)` (resolves the OAuth cred + accountEmail from `config`). One **email reconcile worker per instance** (mirror the WA reconcile worker: `pull(cursor)` every 60s, cursor in `channel_instances.sync_cursor`, advance after ingest). Ingestion → the SAME `ingestInbound` writer (agent_inbox, dedup on `(instance, message-id)`).

### DM6-4 · Store TO/CC for the CC rule
`agent_inbox` has no recipients columns. Add **migration 011**: `agent_inbox.recipients JSONB` (`{to:[],cc:[]}`), written by `ingestInbound` from `InboundMessage.recipients`. (Alternative: read from `raw_metadata` — but a first-class column is cleaner + queryable and other channels leave it null.)

### DM6-5 · CC-only triage rule (core, `triage.service`/routing)
Design contract: **"CC-only emails → context-only (no task) unless the body contains an explicit ask."** In the money-loop, for a `channel_type='email'` row where the founder's `accountEmail` is in `cc` but NOT `to`: suppress task creation **unless** an extracted intent is actionable with `confidence ≥ threshold` (an explicit ask). Implementation: the context loader passes `ccOnly` (computed from `recipients` + the instance's accountEmail) into routing; a CC-only row with only non-actionable intents (`info_provided`/`compliment`/`unclear`) → mark `processed`, record a `context` decision, no task. **Reuses the LLM** — a CC-only FYI naturally yields non-actionable intents; the rule is the explicit guard so a borderline case doesn't spam tasks.

### DM6-6 · Reply-from instance (outbound)
When M1.8 sends an email reply, it goes out through the **instance that received the thread** (the inbox row's `channel_instance_id`); a NEW outbound email uses `agent_customers.default_email_instance_id`. M1.6 only needs `send()` on the adapter (wired to the drainer in M1.8) — build it, leave it unwired.

## 5. DA pre-build questions
1. Gmail **History API gaps** (historyId expiry, missed events between polls) — is bootstrap-on-404 + the 2-day `newer_than` bootstrap window enough, or do we need a periodic full reconcile (change 03)?
2. **CC-only rule** — compute `ccOnly` from stored `recipients` + accountEmail, or lean entirely on the LLM (non-actionable intent → no task)? The explicit guard vs simplicity.
3. **Recipients storage** — first-class `agent_inbox.recipients` column (migration 011) vs reading `raw_metadata`?
4. **Self-sent skip** — `from === accountEmail` only, or also skip our own agent-sent replies (M1.8) to avoid a feedback loop (they'd be `direction=outbound` context rows anyway)?
5. **Source dedup for email** — `sourceEntityId = threadId` (Gmail thread) — confirm it's stable across replies (Gmail keeps threadId stable) so the M1.5b source-unique dedup groups a thread into one task.
