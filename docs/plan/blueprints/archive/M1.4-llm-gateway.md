## Scope boundary (avoid smuggling M1.5b)
M1.4 builds the **LLM machinery** and exercises `extractIntents`/`judgeSimilarity` against a **canned `TriageContext`** (hand-built in `triage:sample`). It does **NOT** build: the context loader (real customer/tasks/thread from DB), routing (create-vs-comment), dedup, task creation, or `agent_decisions` — all **M1.5b**. The triage *prompt* + Intent *schema* + the LLM call live here (needed for the golden-schema + failover gate); consuming the intents is M1.5b.

## 1. Design decisions

### DM4-1 · Migration 009 — `009_credentials_and_llm_costs.sql`
Port `credentials` (+ `set_updated_at` trigger, reused from migration 001). Add per design.md:
```sql
CREATE TABLE llm_costs (
  id BIGSERIAL PRIMARY KEY,
  provider TEXT NOT NULL, model TEXT NOT NULL,
  role TEXT NOT NULL,                                  -- 'triage'|'classify'|'draft'
  customer_id UUID REFERENCES agent_customers(id),     -- nullable (canned samples → null)
  input_tokens INT NOT NULL, output_tokens INT NOT NULL,
  cost_usd NUMERIC(10,6) NOT NULL,
  created_at TIMESTAMPTZ NOT NULL DEFAULT now()
);
CREATE INDEX idx_llm_costs_created_at ON llm_costs(created_at DESC);  -- daily-cap sum
```

### DM4-2 · Sealed store (`src/crypto/secret-box.ts` + `src/config/credentials-store.ts`)
Port `secret-box.ts` (encrypt/decrypt/isEncryptionConfigured). **`CREDENTIALS_ENCRYPTION_KEY` is the one master secret read directly via `process.env`, NOT the zod schema** (it's the KEK that unlocks the store — it cannot live in the store; keeping it out of the zod schema preserves the "no secrets in env schema" invariant). `CredentialsStore`: `load()` (decrypt all → in-memory cache at boot), `get/has/set/list/remove`, encrypted-at-rest, plaintext only in memory, never logged.

### DM4-3 · Resolver upgrade (`src/config/credentials.ts`)
`resolveCredential(ref)` → `store.get(ref) ?? process.env[ref]` then throw-if-missing (unchanged contract for required creds). Add **`tryResolveCredential(ref): string | undefined`** (no throw) for provider keys behind the fallback chain — a missing key must trigger failover, not crash. All existing callers (EZY/WA/WEBHOOK/Telegram) keep working via env fallback. **Canonical ref names are env-var style** (`ANTHROPIC_API_KEY`, `OPENAI_API_KEY`, `DEEPSEEK_API_KEY`, `ADMIN_API_KEY`) — same string is both the store key and the env var, so store and env never diverge.

### DM4-4 · Provider clients (`src/adapters/llm/{anthropic,openai,deepseek}.client.ts`) — raw fetch, implement `LlmProviderClient`
Each: ctor `(resolveKey: () => string|undefined, baseUrl)`; `complete()` + `completeStructured<T>()`; lazy key resolve; retry on 429/5xx (reuse `adapters/shared/retry`); map usage → `{inputTokens, outputTokens}`. A missing key or auth 401/403 throws a typed **`LlmProviderError{kind:'auth'|'rate'|'server'|'schema', provider}`** the router treats as hard-fail → failover.
- **Anthropic:** `POST {base}/v1/messages`, headers `x-api-key`,`anthropic-version:2023-06-01`. Structured = **top-level `output_config:{format:{type:'json_schema',schema}}`** (⟲ **VERIFIED against platform.claude.com docs during build** — this is the current canonical mechanism, NOT forced tool-use). The result lands as a **JSON string in a `text` content block**; parse via `content.find(b=>b.type==='text').text` → `JSON.parse`. **claude-sonnet-5 runs adaptive thinking ON by default and does NOT support `thinking:{type:'disabled'}` or `budget_tokens` (→ 400)** — so send NEITHER; a `thinking` block precedes the `text` block, which is exactly why we locate by type, **never `content[0]`** (R41). usage=`{input_tokens,output_tokens}`. base default `https://api.anthropic.com`.
- **OpenAI:** `POST {base}/v1/chat/completions`, `Authorization: Bearer`. Structured = `response_format:{type:'json_schema',json_schema:{name,schema,strict:true}}`. usage=`{prompt_tokens,completion_tokens}`. base default `https://api.openai.com/v1`.
- **DeepSeek:** OpenAI-compatible; `response_format:{type:'json_object'}` (no json_schema support → schema is embedded in the system prompt + validated client-side). base default `https://api.deepseek.com`.

### DM4-5 · `LlmRouter` (`src/adapters/llm/llm-router.ts`) implements `AgentLlmPort`
Config resolved from env (DM4-6). `callStructured(role, schema, system, messages)`:
1. Order = `[roleProvider ?? defaultProvider, ...fallbackChain]` deduped.
2. **Daily cost cap (R17 — LLM spend ungated; NOT R14) checked FIRST:** `SELECT coalesce(sum(cost_usd),0) FROM llm_costs WHERE created_at >= date_trunc('day', now() AT TIME ZONE 'America/Panama') AT TIME ZONE 'America/Panama'` — ⟲ **R42: pin the day boundary to the founder tz, not the ambient session `TimeZone` GUC** (UTC-midnight ≈ 19:00 Panama, and shifts per connection). If `>= LLM_DAILY_COST_CAP_USD` → throw `CostCapExceeded` + admin-notify **once/day** (kill-switch). ⟲ DA §6-Q1: this is a **soft** cap — worst-case overshoot = `in-flight-calls × per-call-cost` (concurrency 1 in M1.4; bounded by M1.5b's worker-claim later) — documented, not unbounded; a hard cap would need atomic reserve-then-spend.
3. Try each provider: `completeStructured` → **validate value against the schema (zod)**; on `LlmProviderError` (auth/rate/server) OR schema-invalid → record nothing billable? No — record cost for any call that returned usage, then move to next provider + `notifyAdmin('failover provider→next: reason')`. 
4. On success → insert `llm_costs` row (provider, model, role, tokens, `cost_usd` from `pricing.ts`), return value. If ALL providers fail → throw `LlmAllProvidersFailed` (M1.5b's inbox retry holds the work).
- `extractIntents(ctx)` → `callStructured('triage', INTENT_ARRAY_SCHEMA, TRIAGE_SYSTEM, [userFrom(ctx)])` → `Intent[]`.
- `judgeSimilarity(a, candidates)` → `callStructured('classify', SCORES_SCHEMA, SIM_SYSTEM, …)` → `number[]` aligned to candidates.
- **admin-notify** is an injected `notifyAdmin(msg): Promise<void>` (wired to `TelegramNotifier.notifyAdmin` in main.ts) — the router depends on a callback, not the Telegram adapter directly.

### DM4-6 · Config + per-(provider,role) model resolution (`src/config/env.ts`) — ⟲ DA B1
**A single model-per-role is a failover dead-end (R38):** the chain is `[anthropic, openai, deepseek]` but `claude-sonnet-5` handed to OpenAI → 404 → `LlmAllProvidersFailed`. Failover NEVER succeeds. So the router resolves the model by **(provider, role)** — the provider currently being tried picks its OWN model:
```
modelFor(provider, role):  env LLM_MODEL_<PROVIDER>_<ROLE>  ??  DEFAULTS[provider][role]
```
Defaults (env-overridable, since IDs drift): anthropic → triage/draft `claude-sonnet-5`, classify `claude-haiku-4-5`; openai → triage/draft `gpt-4.1`, classify `gpt-4.1-mini`; deepseek → all `deepseek-chat`. **`pricing.ts` keys must match these real per-provider IDs** (else fallback cost = 0). Other env: `LLM_DEFAULT_PROVIDER` (anthropic), `LLM_FALLBACK_CHAIN` (csv, default `openai,deepseek`), `{ANTHROPIC,OPENAI,DEEPSEEK}_BASE_URL` (overridable → tests point at a local mock), `LLM_DAILY_COST_CAP_USD` (10). Provider **keys** are NOT here (resolveCredential). A per-role *preferred provider* override (`LLM_TRIAGE_PROVIDER`, …) defaults to `LLM_DEFAULT_PROVIDER`.

### DM4-7 · Golden Intent schema (`src/adapters/llm/triage-prompt.ts`) — ⟲ DA B3/R40
ONE JSON-schema object (matching `Intent` in llm.port.ts, snake_case) + a **separate zod validator** (range/format) + the triage system prompt. "One schema, three providers" only holds under the **intersection of Anthropic strict tool-use and OpenAI `json_schema` strict**, so the WIRE schema MUST be strict-clean:
- `additionalProperties:false` on **every** object; **every** property listed in `required` (strict = no optionals);
- nullable via **type-union** — `related_open_task_ref: {type:["string","null"]}`, not an omitted optional;
- **NO** `minimum`/`maximum`/`minLength`/`maxLength`/`multipleOf`/`format` — these 400 the strict providers. **`confidence` is the trap**: no `minimum:0,maximum:1` in the wire schema — the 0..1 range check moves to the **zod validator**.
The SAME strict schema feeds all three `completeStructured` calls → schema-valid regardless of provider (gate golden-schema check). **Unit test asserts the schema contains none of the banned keywords.** Zod rejects any non-conforming value → failover; a malformed intent never leaks.

### DM4-8 · Admin endpoints (`src/adapters/admin/admin.router.ts`)
`POST /admin/credentials {name,value}` → `store.set` → 200 `{name,last4,updated_at}` (**value never returned; only last4**); `GET /admin/credentials` → list (last4 only); `DELETE /admin/credentials/:name`. **Guarded by `ADMIN_API_KEY`** (header `x-admin-key`, `crypto.timingSafeEqual` with a **length-guard first** — same footgun as the M1.3 signature verify). **If `ADMIN_API_KEY` is unset the admin router is NOT mounted** (fail-closed — never an unauthenticated mutating surface) and **main.ts logs "admin router not mounted (ADMIN_API_KEY unset)" at boot** so a forgotten key is diagnosable, not a silent 404 (⟲ DA). Mounted in `buildApp` via `AppDeps.adminRouter` **after `express.json()`, before the 404 catch-all**.

### DM4-9 · `pricing.ts`
`{provider: {model: {inUsdPer1M, outUsdPer1M}}}` approximate table + `costUsd(provider,model,usage)`; unknown model → 0 cost + a `logger.warn` (cost visible, never crashes a call). Documented as best-effort (rates drift) — the point is relative spend + the cap, not billing accuracy.

## 5. Yuval's acceptance test (the gate)
**Prereqs Yuval provides:** `CREDENTIALS_ENCRYPTION_KEY` (any strong string) + `ADMIN_API_KEY` in the orchestrator `.env`; **2 real provider keys** (Anthropic + OpenAI or DeepSeek) set via `POST /admin/credentials` or env.
1. **Structured triage + cost:** `npm run triage:sample` on a canned bug-report context → prints schema-valid `intents[]` + a new `llm_costs` row (provider/model/tokens/cost, key `last4` shown, no full key). 
2. **Golden schema across providers:** `npm run triage:sample -- --provider=openai` (and `=deepseek`) → each returns schema-valid intents (same shape, different provider).
3. **Failover:** set a bad Anthropic key (or `--provider` with a revoked key) → the call completes via the next provider in the chain + a Telegram **admin-topic** notice; a valid run afterward uses the default again.
4. **Cost cap:** set `LLM_DAILY_COST_CAP_USD=0` → next `triage:sample` is refused with a kill-switch error + one admin notice (proves R14).
5. **Secret hygiene:** `GET /admin/credentials` shows only `last4`; the DB `credentials` rows are ciphertext; no key in any log.

## 6. DA pre-build questions
1. Cost cap as a **per-call pre-check** (chosen) vs a worker — is the pre-check race-safe enough (a burst could slip a few calls over the cap between the check and the insert)? Acceptable overshoot?
2. Failover on **schema-invalid** output (not just transport errors) — right call, or should a schema failure retry the SAME provider once before failing over (cost vs latency)?
3. `judgeSimilarity` is used by M1.5b dedup — build it now (port-complete) or stub to `throw 'M1.5b'`? (Proposed: build it — it's the same `callStructured` machinery, cheaply tested, and keeps `AgentLlmPort` honest.)
4. Admin router unmounted when `ADMIN_API_KEY` unset — does that break the gate if Yuval forgets the key? (Proposed: yes-by-design; `triage:sample` also reads env keys directly, so the gate works without the admin endpoint — the endpoint is one path to set keys, env is the other.)
5. `CREDENTIALS_ENCRYPTION_KEY` via `process.env` direct (not zod) — agree this is the correct exception to the no-secrets-in-env-schema rule?
