## 1. Design decisions

### DM5b-1 ¬∑ Inbox processor worker (task 6.6) ‚Äî the first REAL worker
`buildInboxProcessorWorker(triageService)` ‚Üí `WorkerDefinition` (interval ~10s). Each tick:
- **Claim** a small batch from `agent_inbox` (CLAIM_TEMPLATE): `status='pending' OR (status='processing' AND updated_at < now()-'10 min')`, `SET status='processing', retry_count=retry_count+1 RETURNING id` (IDs only ‚Äî never the body).
- **Null-body grace (R36):** the claim WHERE also requires `(body IS NOT NULL OR received_at < now()-'10 min')` ‚Äî a voice note lands with `body=NULL` and the transcript fills it on a later M1.3 reconcile; don't triage an empty body until the grace window elapses (then it triages to `unclear`‚ÜíaskFounder). Only `direction='inbound'` rows (skip `skipped` context rows).
- Process the claimed batch **SEQUENTIALLY** (`for ‚Ä¶ of ‚Ä¶ await`, NOT `Promise.all`) ‚Üí concurrency stays 1, so the R43 soft cost-cap holds unchanged (‚ü≤ DA: only concurrent processing would force hardening R43 here). For each claimed id ‚Üí `triageService.process(inboxId)`; on success ‚Üí `status='processed', processed_at=now()`; on throw ‚Üí leave `processing` (reclaimed) ‚Äî the worker-runner records the failure.
- **R49 idempotency short-circuit:** `process()` first checks `agent_tasks` for a row with this `inbox_message_id`; if present, the LLM+action already ran on a prior attempt (that then failed at notify) ‚Üí **skip LLM + dedup + createTask** (no re-spend, no spurious duplicate `contributed_to`), just (re)notify + mark `processed`. Closes the notify-failed-‚Üí-retry re-run.
- **Poison-pill (`failStuck`):** a companion statement flips rows with `retry_count >= 3` still failing to `status='failed', last_error=‚Ä¶` + one **admin-topic** alert. (Runs each tick before/after claim.)
- **Retire the heartbeat worker** now (its own note: remove once the first real worker exists).

### DM5b-2 ¬∑ Context loader (task 6.2, core `src/triage/context-loader.ts`)
For a claimed inbox row: (1) `resolveContact({channelType, address: sender_address})`; `known`‚Üí`customerId`; `unknown`‚Üí mark inbox `skipped` + increment a skipped counter (admin weekly digest is change 03) and STOP; `propose` (email-domain, M1.6 mostly) ‚Üí `proposeAddContact` + STOP. (2) Load `agent_customers` config by `customerId`. (3) `taskTarget.findOpenTasks({projectRef})` ‚Üí open tasks (dedup candidates). (4) Last 10 same-thread messages: `agent_inbox WHERE channel_thread_id=$thread AND direction='inbound' ORDER BY received_at DESC LIMIT 10`. ‚Üí assemble `TriageContext` ({message, customer, recentTasks}). Also backfill `agent_inbox.customer_id` (M1.3 left it null).

### DM5b-3 ¬∑ Intent extraction + routing (task 6.3, core)
`llm.extractIntents(context)` ‚Üí `Intent[]`. Per intent, routing (design triage contract):
- `confidence < 0.5` OR `category='unclear'` ‚Üí `askFounder` (needs-input template), record `agent_decisions(outcome='pending')`, no task.
- else ‚Üí **dedup** (DM5b-4) ‚Üí create or comment.
- (CC-only email rule is M1.6 ‚Äî WhatsApp has no CC.) A message may yield 0..N intents; 0 ‚Üí mark processed, no action.

### DM5b-4 ¬∑ Dedup (task 6.4, core `src/triage/dedup.ts`) ‚Äî ‚ü≤ DA B1/R48 (portal-primary)
The **portal `findOpenTasks(sourceEntity)` (open-only) is the dedup source of truth** ‚Äî durable (survives a lost local bridge row after a mid-crash ‚Üí closes R47's same-call dup) AND status-aware (won't comment on a closed task). The local `agent_tasks` bridge is NOT consulted for dedup (a stale bridge row ‚Üí comment-on-closed/409); it stays purely the audit trail + R49's per-inbox-message idempotency check.
1. **Thread dedup:** `taskTarget.findOpenTasks({projectRef, sourceEntity:{type:channelType, id:threadKey}})` ‚Äî scoped (passes M1.5a's R46 fail-loud guard), open-only via the server `status` filter, ‚â§ a few rows so page-1 is complete ‚Üí if any ‚Üí **comment** on the most recent.
2. **Title similarity:** else `judgeSimilarity(intent.suggested_title, titlesOf(findOpenTasks({projectRef})))` ‚Üí if `max >= 0.8` ‚Üí **comment** on the matched open task. *(‚ü≤ R46 residual: `findOpenTasks({projectRef})` is page-1/‚â§25 ‚Äî a customer with >25 open tasks misses candidates beyond page 1; documented, paginate in change-04.)*
3. else ‚Üí **create**.

### DM5b-5 ¬∑ Action execution (task 6.5, core)
- **create:** `taskTarget.createTask({customerRef:bp_ref, projectRef:project_ref, workItemTypeRef:work_item_type_ref, title:intent.suggested_title, description:<summary + original body + source line>, priority:intent.priority, source:{service:'agent-orchestrator', entityType:channelType, entityId:threadKey, display:<customer/thread>, url:<deep link>}, tags:[category]})` ‚Üí `TaskRef`. Insert `agent_tasks(task_ref, customer_id, inbox_message_id, relationship='created_from')`. Notify (DM5b-6, new-task template).
- **comment:** `taskTarget.addComment(task, <summary + body>)`; insert `agent_tasks(relationship='contributed_to')`. Notify (comment-added template).
- Always: `agent_decisions(decision_type='triage', agent_output=<intents>, outcome='accepted', inbox_message_id, customer_id)`.
- Mark `agent_inbox.status='processed'`.
- **Idempotency:** the row is claimed (`processing`) so no concurrent double-process. A crash between `createTask` and the `agent_tasks`/notify steps is safe because dedup (DM5b-4) is **portal-`sourceEntity`-primary** ‚Äî a re-run finds the just-created task and comments (or the R49 short-circuit skips entirely if the bridge row did land). No reliance on the local bridge for dedup.

### DM5b-6 ¬∑ Telegram notify + deep links (task 7.1)
Templates (to the customer topic via `notifyCustomerEvent`): **new task**, **comment added**, **needs input** (askFounder), **new-contact proposal**, **failure alert** (admin topic). Each task notice carries a **deep link** (`<EZY_PORTAL_BASE_URL>/projects/tasks/<ref>` ‚Äî ‚öÝ verify the portal frontend task URL at build) and an **‚ùå inline button** whose `callback_data` encodes the action + task ref (compact: `x:<task_ref>`; Telegram caps callback_data at 64 bytes ‚Äî a UUID fits). Extend the notifier with a `notifyCustomerEvent(..., buttons?)` path (askFounder already builds `inline_keyboard`).

### DM5b-7 ¬∑ Callback poller + ‚ùå undo (task 7.2)
- **telegram-client:** add `getUpdates({offset, timeout, allowedUpdates:['callback_query']})` (long-poll) + `answerCallbackQuery(id, text?)`.
- **Callback poller worker** (composition): long-poll `getUpdates` from a **persisted offset** (DM5b-8); for each `callback_query` ‚Üí parse `callback_data` ‚Üí invoke the registered `onDecision({notificationRef:task_ref, optionId:'x', by:from.id})` ‚Üí `answerCallbackQuery` (ack the tap; a stale-id failure is swallowed). Advance + persist offset.
- **onDecision handler** (core): `optionId==='x'` ‚Üí `taskTarget.setStatus(task,'cancelled')` + record `agent_decisions(decision_type='human_override', human_override={action:'cancel', by}, outcome='rejected', resolved_at=now())`. **Idempotent (R11 ‚Äî Telegram re-delivers callbacks):** guard on "already cancelled" ‚Äî if a `human_override cancel` decision for this `task_ref` already exists, no-op (no double-cancel, one override record). Post a confirmation to the customer topic.

### DM5b-8 ¬∑ Migration 010 ‚Äî poll offset + indexes (‚ü≤ DA ¬ß6-Q4, thread index)
- `app_state(key TEXT PRIMARY KEY, value TEXT, updated_at ‚Ä¶)` + `set_updated_at` trigger ‚Äî Telegram `getUpdates` offset (`key='telegram_update_offset'`) + the skipped-sender tally (a counter, NOT a change-03 table). Persist stops restart re-delivery (the override guard is the correctness net).
- **Index `agent_inbox(channel_thread_id, received_at DESC)`** ‚Äî the context loader's last-10-same-thread query + the dedup thread lookups would otherwise seq-scan every message (migration 004 indexed only `status`/`customer_id`).
- **`agent_decisions.task_ref TEXT` + partial-unique index `(task_ref) WHERE decision_type='human_override'`** ‚Äî makes the ‚ùå override *atomically* dedup-able (R21/R11) so a re-delivered callback can't write a second override even under a race (the single sequential poller + check-then-act guard is adequate, but this is cheap insurance).

### DM5b-9 ¬∑ Failure paths (real, not aspirational)
- Portal down mid-triage ‚Üí `process()` throws ‚Üí row stays `processing` ‚Üí reclaimed after 10 min ‚Üí retried; `retry_count>=3` ‚Üí `failed` + admin alert (task 9.5 drill).
- LLM all-providers-fail (M1.4 `LlmAllProvidersFailed`) ‚Üí throw ‚Üí same retry lifecycle (the inbox holds the work).
- Restart mid-tick ‚Üí claimed-but-unfinished rows (`processing`) reclaimed by the stuck-clause; the sourceEntity dedup prevents a double task.

An unresolved sender ‚Üí `skip` ‚Üí **the gate is a silent no-op** ‚Äî this is the #1 gate trap.
1. **First message ‚Üí task + notify:** send a WhatsApp message (a bug/request) from the linked contact ‚Üí within ~1 min an **EZY task** is created (verify `GET ‚Ä¶/tasks?sourceService=agent-orchestrator&sourceEntityId=<number>` or the portal UI) **and a Telegram notification** lands in that customer's topic with an ‚ùå button.
2. **Second same-thread message ‚Üí comment (no dup):** send another ‚Üí it **comments on the same task**, no second task (dedup).
3. **‚ùå undo:** tap ‚ùå ‚Üí the task goes `cancelled` + an override is recorded (`agent_decisions`); tap ‚ùå again ‚Üí **no double-cancel** (idempotent).
4. **Unclear ‚Üí needs input:** send a vague message ‚Üí an **askFounder** prompt in the topic, no task.
5. **Failure drill (optional):** stop the portal, send a message ‚Üí the inbox row retries then `failed` + admin alert; restart the portal ‚Üí recovery reprocesses.
