## Architecture decisions for this milestone (things design.md doesn't spell out)

1. **`src/main.ts` vs `src/app.ts` split.** `app.ts` = pure Express factory (testable, no side effects), `main.ts` = env→migrate→listen→shutdown bootstrap. Deliberate improvement over the combined ported pattern.
2. **Row-claiming is a documented SQL template, not a generic helper.** whatsapp_manager's precedent is one bespoke method per table. `src/workers/worker-runner.ts` owns the generic interval/backoff/registry; each concrete worker (inbox processor — M1.5b, outbound drainer — M1.8) writes its own `claimPending*()` following `src/workers/CLAIM_TEMPLATE.md`.
3. **Zero business workers in M1.1, one framework self-test worker.** Inbox/outbound worker bodies don't exist until M1.5b/M1.8. M1.1 registers exactly one trivial `heartbeat` worker (`SELECT 1`) so the framework + `/health` surfacing are proven end-to-end. **DA ruling: KEEP, but mark ephemeral** — removed/disabled once the first real worker exists, so it doesn't linger as permanent log noise.
4. **Placeholder supporting types.** design.md references `TargetTask`, `TargetTicket`, `Notification`, `Intent`, `TriageContext`, `LlmMessage`, `TokenUsage` without defining them. Minimal shapes added now, flagged refine-when-implemented, not schema-authoritative.
5. **`Intent` keeps snake_case fields** (`suggested_title`, `related_open_task_ref`) exactly as design.md's JSON contract — this is the LLM structured-output wire shape.
6. **Migration numbering stays 001–008; 009 (credentials + llm_costs) explicitly out of scope** — owned by M1.4 (tasks.md 6.0).
7. **Seed data lives inline in 001 and 008**, not a separate numbered migration — seeding is forward-only, run-once, identical in nature to DDL.
8. **Defensive `CREATE EXTENSION IF NOT EXISTS pgcrypto;`** in 001 (guarantees `gen_random_uuid()`). Blueprint addition, not a design.md ◆ deviation.
9. **DB bootstrap gap.** `agent_orchestrator` DB doesn't exist yet in `ezy-postgres`. A one-off `npm run db:create` (connects to `postgres` maintenance DB, `CREATE DATABASE agent_orchestrator`, swallows "already exists") runs once before first migrate / `docker compose up`. **◆ BF1: DB host must match the network mode** — under `network_mode: host` (ADR-11) both `db:create` and the app connect to `localhost:42016` (the `ezy-postgres` host-published port), for host-run and container-run alike.
10. **Default `PORT=3100`**, not 3000 — whatsapp_manager already owns 3000.
11. **◆ BF1 — `network_mode: host` on the orchestrator container.** Verified topology: whatsapp_manager runs in its **own** compose with its own Postgres, host-published `3000:3000`, **not on `ezy-network`**; portal-business is a **host process** on `5040` (only nginx-dev is containerised). So neither `whatsapp_manager:3000` nor (from inside a container) `localhost:5040` resolves. `network_mode: host` makes `localhost:3000` (WA), `localhost:5040` (portal), and `localhost:42016` (ezy-postgres host port) all reachable from one coherent address space. **DB env must match host mode** → `PGHOST=localhost PGPORT=42016` (not `ezy-postgres:5432`). Alternative (rejected as more moving parts on this single-user Linux box): stay on `ezy-network` + `extra_hosts: ["host.docker.internal:host-gateway"]` (host-gateway is **not** automatic on Linux) and use `http://host.docker.internal:{3000,5040}`.

**001_channel_instances.sql** ◆ replaces all channel CHECK enums
```sql
CREATE EXTENSION IF NOT EXISTS pgcrypto;   -- blueprint addition: guarantees gen_random_uuid()

-- ◆ BF2: shared updated_at auto-stamp, ported from whatsapp_manager migration 010 (trg_messages_updated_at).
-- Required so the CLAIM_TEMPLATE stuck-row reclaim (status='processing' AND updated_at < now()-interval)
-- measures age from last claim, not row creation. Attached to every MUTABLE table (001/002/004/006).
CREATE OR REPLACE FUNCTION set_updated_at() RETURNS trigger AS $$
BEGIN NEW.updated_at = now(); RETURN NEW; END;
$$ LANGUAGE plpgsql;

CREATE TABLE channel_instances (
  id              UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  channel_type    TEXT NOT NULL,
  provider        TEXT NOT NULL,
  name            TEXT NOT NULL UNIQUE,
  config          JSONB NOT NULL DEFAULT '{}',
  credentials_ref TEXT,

  sync_cursor     TEXT,

  updated_at      TIMESTAMPTZ DEFAULT now()

CREATE TRIGGER trg_channel_instances_updated_at BEFORE UPDATE ON channel_instances
```

## 5. ESLint import-boundary config (eslint.config.mjs)

```js
import tseslint from 'typescript-eslint';
import importPlugin from 'eslint-plugin-import';

export default tseslint.config(
  { ignores: ['dist/**', 'node_modules/**'] },
  ...tseslint.configs.recommended,
  {
    files: ['src/**/*.ts'],
    plugins: { import: importPlugin },
    settings: { 'import/resolver': { typescript: true } },
    rules: {
      'import/no-restricted-paths': ['error', {
        zones: [{
          target: [
            './src/inbox', './src/triage', './src/customers',
            './src/outbound', './src/decisions', './src/ports',
          ],
          from: './src/adapters',
          message: 'Core domain/port modules may not import adapters (D1 — hexagonal core). Wire adapters only in src/main.ts.',
        }],
      }],
    },
  },
);
```
**◆ DA-required: permanent regression guard, not a temp toggle.** The import boundary is *the* load-bearing invariant (D1) — a one-time manual check leaves nothing to catch a later flat-config/resolver breakage that silently disables the rule. Ship a **committed** fixture `test/boundary/illegal-import.ts` containing a real core→adapter import, and a `lint:boundary` npm script that runs ESLint (with this config) over that fixture and **asserts a non-zero exit** (the rule must reject it); wire `lint:boundary` into the build/CI. Because the fixture must be matched by the zone, either (i) place it under a zone `target` path (e.g. `src/inbox/__illegal_import_fixture__.ts`, excluded from `tsc` via `tsconfig` and from `dist`) and assert lint errors on it, or (ii) add `test/boundary` to the zone `target` in a boundary-only ESLint invocation. Builder picks the mechanism that reliably yields a non-zero exit; the M1.1 gate asserts `lint:boundary` fails-closed. **◆ Self-defeating-fixture trap (DA):** the fixture contains a *real* core→adapter import, so it must be **excluded from the default `lint` / `typecheck` / `build`** (e.g. `tsconfig` exclude + `eslint .` ignore) yet still caught by `lint:boundary` with the same rule config. Explicit success condition: **`npm run lint`, `npm run typecheck`, and `npm run build` all stay GREEN with the fixture present; only `npm run lint:boundary` goes RED on it.**

**◆ Guardrail — the zone is allowlist-by-omission:** the `target` array names *today's* core dirs. A **new** core module in a later milestone (e.g. M2 `src/memory/`) is unprotected until manually added to the zone. Every milestone that adds a core dir MUST extend this `target` list — record this in the M2+ blueprints.

## 6. Worker-framework interface (src/workers/worker-runner.ts)

```ts
export interface WorkerDefinition {
  name: string;
  intervalMs: number;
  run: () => Promise<void>;
  maxBackoffMs?: number;               // default e.g. 10 * intervalMs
}

export interface WorkerStatus {
  name: string; intervalMs: number;
  lastRunAt: Date | null; lastSuccessAt: Date | null;
  lastDurationMs: number | null; lastError: string | null;   // error MESSAGE only — never payload
  consecutiveFailures: number;
}

export function startWorker(def: WorkerDefinition): { stop(): void };
export function getWorkerStatuses(): WorkerStatus[];         // read by health.service.ts
```
Implementation: recursive `setTimeout` (not `setInterval`, to allow variable backoff), `.unref()`'d, each tick try/catch updating the registry — never logs `run()`'s internals, only `{ worker, durationMs, ok }`. Row-claiming template (`src/workers/CLAIM_TEMPLATE.md`, from `message.service.ts:597-619`):
```sql
UPDATE <table> SET status='processing', retry_count = retry_count + 1
WHERE id IN (
  SELECT id FROM <table>
   WHERE status = 'pending' OR (status='processing' AND updated_at < now() - interval '10 minutes')
   ORDER BY id ASC FOR UPDATE SKIP LOCKED LIMIT $1
) RETURNING id, ...;   -- select only IDs/metadata needed for the log line, never body
```
◆ BF2: the `updated_at < now() - interval` stuck-row clause is correct **only because** the `set_updated_at()` trigger (migration 001) bumps `updated_at` on the claiming UPDATE. Without it, age would be measured from row creation and crash-recovery would silently break. The trigger is the DRY fix; do not also `SET updated_at=now()` in the template (redundant, and diverges if the trigger is ever the single source).

Backlog: single aggregate query per table (`COUNT(*) FILTER (...)`, `MIN(created_at) FILTER (WHERE status='pending')`), reusing `idx_agent_inbox_pending` / `idx_agent_outbound_pending`. Makes **R22** (5-min SLA observability) measurable from now. `/health` degrades (503) independently — a dead DB reports `status:"degraded"` before backlog fields are attempted (guarded, no throw).
